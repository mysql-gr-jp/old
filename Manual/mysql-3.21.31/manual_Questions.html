MySQL メーリングリストと質問/エラー(バグ)レポートの提供方法

MySQL メーリングリストの購読/購読中止

MySQL リストへの追加または削除の要求は、電子メールアドレス mdomo@tcx.seに送ってください。subscribe mysql または un-subscribe mysql のどちらか1行のメッセージの送信で十分です。も しあなたの返信アドレスが正しくないのであれば、subscribe mysql your@address.your-domain または un-subscribe mysql your@address.your-domain を使用してください。

購読と購読中止に関するメールを mysql@tcx.se に送ることは しないでください。このアドレスに送られたメールは、自動的に 何百という我々のユーザに転送されるからです。

あなたのローカルサイトが多くの mysql@tcx.se 購読者を持つことも あります。この場合、ローカルメーリングリストを作り、tcx.se からの 一つのメッセージがそのサイトに送られ、ローカルなリストに複写されるように してください。この場合、ローカル mysql@tcx.se リストへの追加と 削除は、あなたのシステム管理者に問い合わせて下さい。

mdomo へのメールは majordomo によって自動的に処理されます。

次のメーリングリストがあります:

mysql-announce
これは MySQL と関連プログラムの新しいバージョンのアナウンスの形 式です。これは、全ての MySQL ユーザが入るべきと我々が考える少な い量のリストです。
mysql
一般の MySQL 議論のための主なリストです。専門的なリストに行くべ きものもあることに注意してください。間違ったリストに投稿しても、答を得ら れないでしょう！
mysql-digest
ダイジェスト形式の上のリストです。これは、一日に一回、一つのメールで全て のメッセージが得られることを意味します。
mysql-Java
MySQL と JAVA についての議論。多くは JDBC ドライバについてです。
mysql-win32
Windows/NT のような Microsoft OS 上の MySQL に関連する全て。
myodbc
ODBC で MySQL への接続に関連する全て。
msql-mysql-modules
MySQL でサポートする Perl についてのリスト。
msql-mysql-modules-digest
上記のダイジェストバージョン。

上述したのと同じ方法で全てのリストを購読/購読中止します。mysql を リスト名に変更してください。

質問またはバグ報告

メーリングリストへ質問を尋ねる前に、このマニュアルを調べるのは良い考えで す。これに答が見つけられない場合は、ローカルな MySQL 熟練者とと もに調べて下さい。まだあなたの質問の回答が見つけられなければ、先に進み、 次のセクションの mysql@tcx.se へメールを送る方法についてを読ん でください。

バグを見つけたと思う。私を助けるためにあなたが必要な情報は？

できれば、配布内の scripts ディレクトリ内にある `mysqlbug' スクリプ トを使ってください。それができない場合は、(適切な)次の記述をすることを覚 えておいてください。情報が多すぎる手紙には回答できますが、少なすぎるとで きないということに注意してください。質問が、使用している MySQL バージョンに関連している場合は、あなたは常に `mysqlbug' を使用する べきです！

mysqlbug は次の情報の多くを自動的に見つけます。しかし、重要な何かが足り ない場合は、あなたの質問にそれを含めてください！

  1. 使用している MySQL のバージョン (例えば mysql-3.21..tgz)。どの バージョンを実行しているかは mysqladmin version の入力で見つけら れます。
  2. あなたが作業しているマシンのメーカーとモデル。
  3. オペレーティングシステム。多くのオペレーティングシステムでは、 uname -a からこれを得られます。
  4. 時にはメモリ(実メモリと仮想メモリ)の量も関係します。
  5. これがコンパイル時のバグの場合: 正確なエラーメッセージと、エラーになった ファイルの問題のコードの回りの数行も含めて下さい。
  6. リファレンスマニュアルとメールアーカイブをチェックしたことをメールに記述 してください。あなたが自分で自分の問題を解決しようとしたことを他の人が知 るためです。
  7. これが実行時のバグの場合、どのようにしてエラーになったかを正確に述べて下 さい。もしエラーになるテストプログラムとテーブルを含めることができるのな ら、おそらくさらに正確な回答が得られるでしょう。
  8. 数行のテストケースを提供できない場合、またはテストテーブルが全員にメール を送るには大きすぎる(10行以上)場合、あなたのテーブルの mysqldump とあな たの問題を説明した README を作ってください。ファイルを tar と gzip また は zip して、そのアーカイブを ftp://www.tcx.se/pub/mysql/hidden に ftp してください。この後、短い問題の説明を mysql@tcx.se に送ってく ださい。
  9. 質問が特権システムに関連する場合、mysqlaccess の出力、'mysqladmin reload'、接続しようとした時に得られた全てのエラーメッセージを含めてくだ さい！ 上述の順でテストを行うべきです！

あなたがサポート顧客なら、より高い優先順位で取り扱うため、特別なメーリン グリストに投稿してください。

回答があなたに個人的に送られて、メーリングリストに送られていない時、回答 を要約し、それをメーリングリストにメールすることは、良いエチケットです。

MySQL keeps crashing

何かがクラッシュする理由を知るのは非常に難しいので、次の一つを行ってくだ さい。

他で働くものがあなたではクラッシュする場合、最初に次のチェックを試みてく ださい:

  • ベンチマークを試しましたか？ これは MySQL をより良くテストしま す。あなたのアプリケーションをシミュレートするコードを追加することもでき ます。
  • fork_test.pl と fork2_test.pl も試してください。
  • mysqld のデバッグについてリファレンスマニュアルをチェックしてください。
  • configure に --with-debug スイッチをつけて MySQL を再コ ンパイルしてください。これは、いくつかのエラーを発見できる安全なメモリ割 り当てを含みます。これは何が起きているかについての多くの出力も与えます。
  • mysqld --log を使用して、いくつかの特有のクエリがそれを殺しているかどう かを見つけることを試してください。全てのバグの 95% は特有のクエリに関係 しています！
  • あなたの OS の最新のパッチを適用しましたか？
  • --skip-locking を使用してください。
  • 死んだときに 'mysqladmin proc' を試しましたか?
  • いくつかのウィンドウで、統計を出力する次のコマンドを行ってください: mysqladmin -i 5 status
    1. gdb (または他のデバッガ) で mysqld を開始してください。
    2. あなたのテストスクリプトを実行してください。
    3. mysqld がコアダンプした時、back (またはあなたのデバッガの backtrace コマンド) を行ってください。
  • MySQL にクラッシュ/間違った振る舞いをさせるあなたのアプリケーショ ンのシミュレートを perl スクリプトで試みてください。
  • または通常のバグレポートを送ってください。 「バグを見つけたと思う。私を助けるためにあなたが必要な情報は？」節参照 。しかし通常 よりもさらに詳細にしてください。MySQL は多くの人のために働いて いるため、クラッシュはあなたのコンピュータだけに存在する何かから来ている と思われます。例えば、システムライブラリのエラーなどです。

メーリングリストで質問に答えるためのガイドライン

オリジナルの投稿者よりも広範囲の人々がそれから恩恵を受けるように回答を作 成することを試みてください。あなたの回答が広く関心を持たれると考えられる 場合、尋ねた人に個人的に直接返事をする代わりに、メーリングリストに投稿し たいと思うでしょう。この場合は、あなたの回答が以前の回答と重複していない か確認して下さい。

あなた返事の中で、質問の本質部分の要約を試みてください。質問全体を引用す ることを義務と感じないで下さい。
