私は特に持ってませんが、一応ユーザ会として聞いてきた範囲で
フィードバックを。黒丸は公開質問、白丸は個人的に聞いたもの、
足りないところがあったら補足してください＞とみたさん、たてやまさん
（実際は Monty の話がかなり長く、討論会の時間は大幅に短縮されました）

●開発環境は？
→DDD on linux

● MySQL CC で、より国際語化の進んでいる GTK とかの選択肢はないのか？
→国際語化とマルチプラットフォームのため Qt 以外も検討。
（特に Mac OSX で Qt の見た目が悪い）

●カラムごとにキャラセットが変えられる必要は少ないと考える。遅くなるのでは？
→スピードデメリットがないから実装した。

●MySQL の高機能化は Oracle をひっくり返すのが目標か？
ユーザのニーズがある限り機能を追加する。Oracle ユーザも
使いたければ使えばよい。

●シェアウェア作者のためのライセンス設定はないのか？
→個別に対応したい。ソースコードを MySQL にくれるだけでもよい。

○MySQL 以前に使っていた DB は？
→ ８４年から９５年までこれまた自分で書いた「unireg」。
　MySQL は c++ の勉強もしたかったので、初めて c++ で書いた。
　最初は msql の API が優れていたのでこれと互換にした。
　徐々に SQL 標準を取り入れた。
　MySQL 4.0 のコードはは９５％程０から書き直している。

○将来的に ujis, sjis のサポートはやめるのか？
→そんなつもりはない。

○UTF-8 での日本語フルテキストサーチは実現されるのか？
→もちろんだ。だが、単語の切り出しが難しくもう少し待ってくれ。

○同様にソートは？
→日本語の場合ソートテーブルがかなり大きくなり、苦労している。
　これも鋭意対応中である。

○UTF-8 では日本語は３バイトになるが、mysql コマンドラインから
　使うと２バイト分しか取らないようで画面が乱れる。
→mysql コマンドの実装はまだ完全でない。内部の処理は問題ないはず
　なので他のインターフェースから使ってくれ。

○今からプロジェクトを起こすならどのバージョンで検討すべきか？
→5.0 だ。既にソースは公開しているので使って欲しい。
　我々の「アルファ」はバグがあるからではなく、仕様が変更される
　という意味であり、公開しているものはアルファであっても十分
　動作は安定しているはずだ。また早期に使ってもらうことで
　フィードバックも増え、完成も早まる。

○バージョンが上がって高機能になってもますます高速化しているが
　その秘訣は？
→コードのすべてを自分で把握しているからだ。一ついいアイディアが
　あればそれを全体で使える。MS などにそれが出来ないのは開発チームが
　大きすぎ、コミュニケーションが取れてないからだろう。
