MySQL で使う場合はそうなるでしょうが．．．

元の発言では

>> 友人が博物館だか美術館の人で、そこにあるデータをデータベースで
>> 管理したいと相談してきました。
>> 
>> 3MB程度のjpg画像がおよそ10万枚と、それらに関する文字情報。
>> 
>> とのことで、それらを
>> 
>> 年代やキーワードなど、いくつかの項目で検索→画像を表示→印刷
>> 
>> という流れで利用することを実現したいと思っているとのことでした。
>> 今は実験的にFileMakerを試しているけども1000枚当たりで動かなくなったと。

という訳で、その博物館だか美術館の人が DB(MySQL) に BLOB で
格納するのか、フォルダを作ってファイル管理するのか、なんてー
話は一切してないですよ。

というより、失礼ながら、DB の事が良く分からないのでしょう。

大抵、クライアントとしては「多少とろいのはともかく動けばいい」
ので、画像の格納方法は実際に担当するＳＥが考えることです。

> それから、１０万点、という数のことは気にした方が良いポイントなので
> 普通の（？）こととは違うんだと強調しておきたいと思います。

データを突っ込むだけで面倒ですからね。

運用もよく考えないと、SQL のチューニング程度ではどうにもなら
ないでしょう。

PC、ではなく PC サーバといわれる製品や、それでも能力が足りな
ければクラスタリングするとか、もっと上(汎用機)を使うとか．．．。

> 確かに、内容を考えると、目録を作りたいのかもしれなくて、
> それでは、「多分検索はしない」という見方は当たっているかもしれません。

それは上記の方が一体どういう運用を念頭に置いているのか、不明
ですから推測してもしょうが有りません。しかし、少なくとも

>> 年代やキーワードなど、いくつかの項目で検索→画像を表示→印刷

と言ってますから、考慮はしておかないといけないでしょう？

ＳＥとなる方が担当者と話してみて、本当は目録だけでいいという
のならそれで作ればいいですが、それだったら DB にする意味が
あまり無いんじゃないんですかね？
来館者にみせるにもそうですし、自分が研究で使うにしても10万点
を検索機能なしで使うのは大変じゃないでしょうかね。
